# Batch 17: 模型注册可编辑与显示控制 — 需求文档

## 目标

补全 AI Companion 模型注册系统的编辑能力，并在 Appearance 设置中增加消息与指标的可见性控制项。

## 背景

当前已注册模型仅支持切换、重置 token 和删除，不支持直接修改已注册配置。用户若要调整已注册模型的显示名、端点、部署名或模型名，只能删除后重新注册，操作成本高且容易丢失当前切换关系。

同时，当前界面缺少对用户消息和指标信息的显示控制，用户无法根据评审、演示或录屏场景按需隐藏部分界面内容。

## 需求

### 1. 已注册模型支持修改

1. 在 Registered Models 表格中为每条已注册模型增加 Edit 操作。
2. 点击 Edit 后，复用现有 Add Model 表单进入编辑模式，不新增第二套弹窗。
3. 编辑模式下表单需自动回填对应 provider 的配置内容。
4. 编辑模式下 provider 保持只读，若需改为其他 provider，应新增或重新注册模型，避免跨 provider 凭据歧义。
5. 保存后应更新原模型记录，而不是新增一条重复记录。
6. 若 API Key 输入框留空，则保留原有 SecureStorage 中的密钥，不强制用户重新输入。
7. 若编辑后的模型标识发生变化，应正确替换旧记录并避免产生重复冲突。
8. 若被编辑的是当前激活模型，保存后应同步刷新当前 provider 状态、模型名称显示和 token 视图。

### 2. Appearance 设置增加显示控制

1. 在 Appearance 设置中新增 `是否显示用户消息` 开关。
2. 关闭后，历史聊天窗口中的用户消息气泡应隐藏，仅保留 Agent / Companion / 系统相关内容。
3. 在 Appearance 设置中新增 `是否显示 metric 信息` 开关。
4. 关闭后，应隐藏消息相关 metric 信息与 AI Companion KPI / metric 展示区域，避免界面出现空白占位。
5. 两个开关均需持久化到 localStorage，刷新页面后保持生效。

## UI / UX 约束

- 编辑模型应沿用现有 Add Model 区域，不新增额外层级。
- 编辑状态应有明确标题和按钮文案区分，例如 `Edit Model` / `Save Changes`。
- 取消编辑后，表单应恢复到新增状态。
- Appearance 新开关应与现有外观设置同风格、同交互模型。

## 涉及模块

| 模块 | 变更 |
|------|------|
| `index.html` | 模型编辑复用表单文案支持；Appearance 增加两个新开关 |
| `src/ai/aiCompanion.js` | 新增已注册模型读取/更新/编辑事件链路；注册表格增加 Edit 动作 |
| `src/core/application.js` | 模型表单新增 create/edit 双模式；Appearance 开关事件与持久化 |
| `css/components/modals.css` | 注册模型表格 actions 扩展；编辑态样式 |
| `css/responsive/mobile.css` | 小屏下表格 actions 和编辑表单适配 |

## 验收标准

1. 已注册模型行出现 Edit 按钮。
2. 点击 Edit 后表单自动展开并回填对应配置。
3. 保存后模型列表原地更新，不产生重复条目。
4. API Key 留空保存时仍可继续使用原有密钥。
5. 当前激活模型编辑后，顶部状态和当前模型名称同步刷新。
6. Appearance 中可看到 `显示用户消息` 与 `显示 metric 信息` 两个开关。
7. 开关即时生效并在刷新后保持状态。
