# Batch 5: 文档补全与导出模式统一 — 设计方案

## 设计目标

1. 让架构文档与当前实现一致（尤其 DirectLineService 基线）。
2. 让核心模块导出风格一致，降低导入混乱。
3. 让关键文件导入顺序一致，提升可维护性与可读性。

## 任务 A: 文档补全

### A1: DirectLineService 架构文档

交付一份“可回顾、可落地、可扩展”的架构文档，至少覆盖：

1. 服务职责边界
2. MessageEntry 数据模型
3. 事件契约
4. application 接线方式
5. 边界约束与后续扩展点

### A2: Conversation-Aware Thinking 文档

以现有实现为基线补全文档，重点描述：

1. 核心流程
2. 使用场景
3. 关键限制和注意事项

### A3: 索引更新

更新 docs 导航入口，使“当前有效文档”优先可达。

## 任务 B: 导出模式统一

### 统一策略

1. 核心模块采用单一主导出（必要时保留辅助导出）。
2. 工具函数集合允许多导出，但需有清晰职责边界。
3. 数据常量文件允许多导出，但需有稳定命名约定。

### 执行重点

优先处理核心入口和高频依赖模块，避免一次性全量改造带来导入链路风险。

## 任务 C: 导入顺序修正

### 顺序规范

1. utils
2. services
3. managers
4. core/components/ui

### 执行原则

1. 仅调整顺序，不混入功能改动。
2. 每次调整后立即做模块加载验证。

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 导出变更导致导入断链 | 变更后执行全局引用核验 |
| 文档与实现再次偏离 | 以 Batch 3 完成态为基线逐项核对 |
| 导入顺序调整误伤 | 分文件小步提交并即时验证 |
