# Batch 6: 消息性能指标增强 — 设计方案

## 设计目标

在不改变消息业务语义的前提下，为每条 Agent 回复增加可解释、可对比的响应时序指标。

## 指标口径

1. TTFT（Time To First Token）
   - 从用户发送动作确认到第一段有效 Agent 响应到达的时间。
2. TTLT（Time To Last Token）
   - 从用户发送动作确认到最后一段有效 Agent 响应收口的时间。
3. Total Duration
   - 从用户发送动作确认到消息渲染完成的总耗时。

## 采集策略

### 起点统一

- 发送时在 application 层记录唯一发送时间戳。
- 同一轮消息使用同一时间基准，避免跨模块口径漂移。

### 过程采集

- DirectLineService 负责在活动接收链路中判断首段与末段时间。
- MessageEntry 用于承载每条消息的指标字段，避免全局变量污染。

### 终点采集

- 渲染层在消息“视觉完成”时回写完成时间。
- 由统一格式化层输出给消息元信息区。

## 展示策略

1. 指标显示在现有元信息行，不额外增加视觉噪声。
2. 时间单位自适应：短时显示 ms，长时显示 s。
3. 缺失值不强行展示，避免误导用户。

## 边界场景

1. 流式消息多片段：仅首段计 TTFT，收口段计 TTLT。
2. 非流式消息：TTFT 与 TTLT 可能接近。
3. 中断消息：允许指标不完整，并标注为异常完成。

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 指标来源分散导致口径不一致 | 强制以 MessageEntry 为唯一承载对象 |
| 流式结束判定不稳 | 统一由完成事件收口，不依赖单点字符串判断 |
| UI 信息过载 | 仅展示核心三项，保持可读性 |
