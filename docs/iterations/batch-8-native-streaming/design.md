# Batch 8: Copilot Studio 原生 Streaming 输出 — 设计方案

## 设计目标

在不破坏当前稳定聊天链路的前提下，先确认“是否可用”，再决定“如何接入”。

## 设计原则

1. 调研先行: 无官方能力结论前，不修改生产消息链路。
2. 渐进落地: 先做旁路验证，再切主路径。
3. 可回退: 接入后可一键退回现有模拟 streaming。
4. 指标连续: 不破坏 TTFT/TTLT/Duration 的统计口径。

## 当前基线

- 通信层: DirectLineService
- 消息模型: MessageEntry
- 渲染层: 现有模拟流式输出

## 接入设计（若官方确认支持）

### 能力识别层

1. 明确官方活动形态（增量 message、event、typing 组合或其他协议）。
2. 定义“原生 streaming 可用”判定条件。
3. 记录“不可用时”的明确判定条件，避免误判。

### 通信层扩展

在 DirectLineService 内新增原生流式处理分支：

1. 片段到达时更新 MessageEntry 文本并标记 render.status=streaming。
2. 片段完成时标记 isComplete=true，并发送完成事件。
3. 保留去重与错误处理逻辑，避免并发更新重复渲染。

### 渲染层适配

1. messageRenderer 支持同一消息多次增量更新。
2. 首段到达时立即显示，减少首字延迟感知。
3. 结束时统一收口状态，避免“卡在流式中”。

### 回退机制

提供配置开关：

- 开启: 使用原生 streaming 路径。
- 关闭或不可用: 回退模拟流式路径。

## 交付产物

1. 调研结论文档（支持/不支持 + 证据链接 + 版本范围）。
2. 接入设计决策记录（为何采用当前事件映射）。
3. 验证报告（稳定性、指标、兼容性）。

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 官方行为与文档不一致 | 增加真实会话抓包验证并留存证据 |
| chunk 顺序不稳定 | 基于 activity 标识做顺序与去重策略 |
| 指标计算口径漂移 | 保留统一时间戳来源并对比改造前后 |
